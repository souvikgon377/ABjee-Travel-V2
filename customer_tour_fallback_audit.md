# Audit Report: Customer-Facing Tour Places Fallback Mechanism

## Executive Summary
The customer-facing tourist place API (`/api/places`) implements a high-performance, multi-layered caching and fallback strategy. Unlike the Admin panel's broad fallback, the customer side is optimized for scale, utilizing **Request Coalescing**, **Stale-While-Revalidate (SWR)**, and **Indexed Firestore Queries** to ensure sub-100ms response times for the majority of users while maintaining 100% uptime during service degradations.

---

## 🏗️ Architecture Overview

### 1. Tier 1: L1 In-Memory Cache (Application Layer)
*   **Mechanism**: A local `Map` object within the Node.js runtime.
*   **TTL**: 10 seconds (for ultra-hot keys).
*   **Purpose**: Protects against massive concurrency spikes (e.g., social media mentions) where thousands of requests hit the same query in seconds.

### 2. Tier 2: L2 Redis Cache (Distributed Layer)
*   **Mechanism**: Redis-backed storage with versioned keys.
*   **Logic**: Uses **SWR (Stale-While-Revalidate)**. If data is stale but not expired, it returns the stale data immediately and triggers a background refresh via a non-blocking `void` promise.
*   **Locking**: Uses a Redis lock (`lock:swr:*`) to ensure only one background refresh happens per key at a time.

### 3. Tier 3: Discovery Layer (Redis Search Indices)
*   **Mechanism**: Uses custom Redis sets (`idx:test:token:*` and `idx:test:prefix:*`).
*   **Logic**:
    1.  Intersects tokens (`sinter`) to find matching document IDs.
    2.  Falls back to prefix matching if no token matches.
    3.  Fetches ONLY the discovered IDs from Firestore.

### 4. Tier 4: Firestore Indexed Fallback (Failover Layer)
*   **Mechanism**: Triggered if Redis discovery fails or returns zero results.
*   **Logic**: Performs highly efficient **Prefix Queries** using Firestore indexes on `location_search`, `location_lower`, and `name_lower`.
*   **Benefit**: Much faster and cheaper than the Admin side's 2,000-doc scan.

### 5. Tier 5: Browser/CDN Layer (HTTP Cache)
*   **Mechanism**: **ETags** (MD5 hash of results) and `304 Not Modified` responses.
*   **Logic**: If the data hasn't changed, the server returns 0 bytes of payload, and the browser uses its local copy.

## 🔍 Detailed Audit Findings

### ✅ Strengths
*   **Thundering Herd Protection**: The `inFlight` request coalescing map prevents multiple concurrent cache misses from firing duplicate Firestore queries.
*   **Stateless Security**: Pagination cursors are **HMAC signed** (`SignedCursorToken`). This ensures users cannot "guess" or manipulate pagination URLs, even if the cache is cleared.
*   **Performance First**: The SWR implementation ensures that the user *never* waits for a slow Firestore query if any cached version exists.
*   **Versioned Invalidation**: `getCacheVersion()` allows for instant global cache busting across all tiers (L1, L2, CDN) during deployment or data migrations.

### ⚠️ Risks & Recommendations

| Finding | Impact | Recommendation |
| :--- | :--- | :--- |
| **L1 Cache Leak** | Medium | The `L1_CACHE` Map does not have a global size limit or eviction policy (LRU). | Use a library like `lru-cache` or implement a `memoryEvict` strategy similar to `hybridCache.ts`. |
| **Redis Discovery Dependency** | Low | If Redis is down, it falls back to Firestore Prefix queries. If those are slow, there is no "Tier 2" local snapshot like the Admin panel. | Consider implementing a small "Hot Snapshot" on disk for the top 100 most popular queries. |
| **Pagination Expiration** | Medium | Cursors are tied to a `queryHash`. If search parameters change slightly, the cursor expires. | Improve UX by providing a "Reset Search" button when the `reset: true` payload is returned. |
| **SWR Lock TTL** | Low | SWR lock is 10 seconds. If a Firestore query takes longer, multiple refreshes might trigger sequentially. | Increase SWR lock TTL to match `SOURCE_FETCH_TIMEOUT` (approx 30s). |
